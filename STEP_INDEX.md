### Known Stale Handoff Issue (cautionary tale)

**2026-05-07**: 새 채팅 시작 시점에 ZIP 안의 HANDOFF.md가 *stale*했음. STEP UX-1 직후 작성된 핸드오프가 그 뒤 진행된 **Document Lifecycle Clarity STEP**을 반영하지 않아 "Invoice Document Activity Layer"가 *다음 작업 후보*로 적혀있었으나 실제 ZIP에는 이미 5개 lifecycle 컴포넌트 + helper + Invoice/Contract wiring이 모두 정착돼 있었음. baseline도 150 kB가 아닌 154 kB.

**근본 원인**: HANDOFF.md는 *작성 시점의 snapshot*인데 그 후 진행된 STEP이 핸드오프에 반영되지 않은 채 ZIP만 새로 묶임.

**대응**: 본 STEP_INDEX.md 도입. 새 채팅에서 작업 시작 *전*에 본 문서 + ARCHITECTURE.md 마지막 entry + 실제 src tree (`ls src/components/document-lifecycle/`)를 교차 확인하여 stale 여부 검증 의무.

### 중복 가능 (진행 시 *기존 위에 확장*)

다음 시스템은 *기존 위에 layer*하는 방향으로 진화 — 절대 *대체 / 재작성* 금지:

- **RBAC** (`src/lib/rbac.ts`) — STAFF / MANAGER / OWNER + ROLE_RANK + hasPermission. STEP 101+ Approval은 RBAC을 *대체하지 않고 위에 layer*. (AXVELA_TRUST_LAYER §1.2 핵심 원칙)
- **승인 대기 queue** (Sidebar) — Contract REVIEW/APPROVED + Settlement non-COMPLETED + Tax non-ISSUED 이미 부분 derive. STEP 109 시 filter / sort / role-based visibility 위에 추가.
- **TimelineEvent** — 기존 actor / kind / detail 구조 그대로. 새 STEP에서 새 kind 추가 시 기존 emit 위에 추가만.
- **Audit category** — 현재 5개 (image_storage / backup / restore / permission / system). STEP 102 시 6번째 (`approval`) 추가.
